1. Executive Summary
--------------------
AR-local is a functioning, valuable single-operator system with genuinely
senior-grade reliability mechanisms - atomic payload install, content-hash
pinning, schema/min-version gates, offline-first behavior, a bounded mobile
worker pool, a cache write-chain mutex, and cross-language crypto interop tests -
embedded in an architecture that grew by feature accretion rather than deliberate
ownership boundaries.

The single most important architectural truth, and the one neither cost nor
structure alone surfaces:

  AR-local must preserve a COMPLETE, PERMANENT, APPEND-ONLY CDR LEDGER from
  2026-05-13 onward, forever. Everything else - dashboards, compact payloads,
  rolling release assets, mobile caches, query windows - is a DERIVED SERVING
  LAYER over that ledger. The system today *intends* permanence but does not yet
  *enforce* it. That gap is the top priority, and it governs how every other
  performance fix must be done (you may prune derived layers; you may never touch
  the ledger).

The causal chain that explains most of the rest, traced end to end:

  The dashboard server's domain logic lives inside one make_handler closure
  (cdr_dashboard_server.py:628), so it cannot be imported -> to reuse ribbon math
  the payload builder copy-pasted it and labeled the copy a "faithful port"
  (app_payload.py:135) -> a recent "rank by comparison rate everywhere" change
  updated ONE copy and not the other -> the web dashboard now aggregates on the
  HEADLINE rate and the mobile payload on the COMPARISON rate. Same product, same
  day, same section, two different "lowest rate" numbers, shipped today. The
  duplication is not a style nit; it is a divergence engine, and it has fired.

Classification:
  - Operational status: functioning single-operator system.
  - Engineering quality: startup-grade, with several production-grade components.
  - Scalability: poor on historical data paths; ledger durability unproven.
  - Branch readiness: not merge-ready (mobile typecheck + zero-warning lint fail;
    see P1-gates for an important caveat on attribution).
  - Rewrite: NO. A rewrite would discard working domain knowledge and reliability
    work.
  - Human-led architectural repair: immediate, staged, ledger-first.


2. The Permanent CDR Ledger Invariant (governing constraint)
------------------------------------------------------------
Owner directive, treated as an architectural invariant rather than a configurable
retention preference:

  - Ledger epoch: 2026-05-13. The ledger grows prospectively, forever.
  - Live state (2026-06-15): 33 daily SQLite databases through 2026-06-15,
    ~6.1 GB retained.
  - 2026-05-14 is MISSING. It must remain an EXPLICIT gap. It must NEVER be
    fabricated by storing a later CDR response under the earlier date.
  - Every finalized daily source snapshot and _exports SQLite is immutable,
    append-only ledger data: never deleted, overwritten, rewritten, or
    destructively migrated.
  - Corrections are APPENDED as explicit revision/correction records; the original
    bytes remain preserved.
  - The dashboard's 90-day limit is a QUERY/READ WINDOW only - never a retention
    bound. "All" must remain able to cover the complete ledger via compact
    aggregates.
  - Derived serving layers - caches, compact history artifacts, rolling GitHub
    release assets, mobile payloads - MAY be bounded, pruned, or regenerated,
    because they are reproducible from the ledger. Their policies must never affect
    ledger retention.
  - The one-SQLite-per-day partition layout is a sound immutable-partition model.
    Do NOT collapse it into a single permanently-growing mutable SQLite file as the
    sole ledger. Build a catalog + derived query layer over the daily partitions.

Why this is currently UNGUARANTEED (code-grounded):

  [SOURCE] cdr_daily.py:99-137 finalizes a day with a completion MARKER and an
    atomic copytree into the persistent export root, and refuses the marker on a
    zero-rate export (132-137). That is finalization-by-flag, not immutability.
    Nothing rejects a later delete or overwrite of a finalized day.

  [SOURCE] cdr_daily.py --force re-runs ingest for a given date; run_ingest fetches
    CURRENT live CDR endpoints and copytree_atomic OVERWRITES that date's
    persistent export root. Concretely: `cdr_daily --force --date 2026-05-14` would
    write today's rates under the missing day - the precise fabrication the owner
    directive forbids - with no guard preventing it.

  [SOURCE] cleanup_removed_cdr_sector.py:26-45 iterates YYYY-MM-DD run directories
    and shutil.rmtree / unlink them, and DROP TABLEs sqlite (18-21). The
    destructive machinery to walk and delete finalized partitions already exists in
    the repo. It is currently scoped to a retired non-banking sector, but a future
    "cleanup" refactor, a wrong argument, or a bug can reach ledger bytes because
    there is no append-only enforcement standing in the way.

  [STRUCT] There is no per-day integrity manifest (hashes, row counts, source
    metadata, previous-day linkage), no continuity verification, no independent
    off-device replication, no tested restore, and no deletion/mutation alerting.
    The Pi is effectively the single copy of the only historical truth.

  [SOURCE] The dashboard history path synthesizes carry-forward rows for missing
    days at serve time (cdr_dashboard_server.py:929 fill_history_gaps). That is
    acceptable as a SERVING-layer concern, but it is a latent fabrication hazard:
    it must never feed back into stored ledger data, and the 2026-05-14 gap must
    surface AS A GAP in any integrity/continuity view, not as a carried-forward
    phantom.

Required correction (P0; see Phase 0A):
  - Make finalized daily partitions append-only; reject mutation, deletion, AND
    same-date overwrite (force re-runs append a revision; never overwrite; never
    write current data under a historical date).
  - Generate and verify per-day integrity manifests + continuity chain from epoch.
  - Maintain >=3 copies incl. one off-device and one independently recoverable;
    run scheduled restore drills.
  - Gate every destructive historical utility behind an audited break-glass step.
  - Alert on missing/changed/unreadable days; preserve 2026-05-14 as a gap record.


3. Reconciliation Across the Three Audits
-----------------------------------------
Confirmed by all three and retained:
  - Core financial semantics and payload contracts are duplicated across server,
    publisher, dashboard, and mobile.
  - app_payload.py, cdr_dashboard_server.py, dashboard/app.js, and
    mobile/src/data/store.ts are oversized ownership failures.
  - Dashboard historical analytics ship raw rows instead of a data product.
  - Per-product mobile history repeats work that belongs on the Pi.
  - Contract validation is inconsistent; tests assert behavior but not cost.
  - WORKFLOW.md is empty while rules call it authoritative.

Corrections accepted (the Opus draft overstated these; corrected here):
  - The server is NOT "untestable." It is integration-testable; the real defect is
    that nested closures are hard to UNIT-test and impossible to import/reuse -
    which is what forced the copy-paste.
  - Broad try/except is not inherently wrong. Ingest/publish use deliberately
    non-fatal optional work. The real failure is that required FRESHNESS and
    optional DEGRADATION are not consistently distinguished or surfaced - a
    successful ingest does not prove the mobile payload is fresh.
  - The if/elif route table (~25 linear comparisons) is NOT a meaningful
    performance problem. Down-ranked from "fix" to "falls out of decomposition."
    (Caveat: it does carry a latent ordering hazard - a bare
    `except FileNotFoundError: pass` mid-chain at cdr_dashboard_server.py:1178 -
    so a dispatch table done right is not LESS safe than the chain, contra a strict
    reading of "could weaken path-safety." Still low priority.)
  - The five download wrappers are low severity. A generic downloadJson<T> is only
    worthwhile if it centralizes runtime validation + metrics.
  - TODO-comment count is a weak signal, not evidence. Down-ranked.
  - The .tmp / _pi_ / _wt- / runs / .ar-* directories are GITIGNORED
    (.gitignore:1-74) - local workstation hygiene, not shipped-repository defects.
  - Issue #0007 was NOT demonstrated to be caused by warn-and-continue; withdrawn.

Sharpened / corrected beyond all three drafts:
  - The ribbon "faithful port" is NOT a future drift RISK (as ranked P1 in the
    ChatGPT draft) - it is a present, shipped, USER-VISIBLE divergence (P0-ribbon).
    The dashboard renders the server's headline-rate aggregate verbatim
    (dashboard/app.js:874 fetches /api/banks/ribbon; ribbonChartItems at app.js:
    498-535 renders range/providers/counts), while mobile builds the same ribbon on
    comparison_rate. "May be a valid product choice" is incorrect: two surfaces of
    the same market showing different distributions is an inconsistency bug, not a
    choice.
  - The mobile refresh guard is NOT race-free (contra the ChatGPT rebuttal of the
    Opus "interleave" claim). store.ts:366-375 checks get().refreshing, then awaits
    onWifi() at line 368 BEFORE setting refreshing:true at 375. Under
    wifiOnly && !manual, two scheduled refreshes can both pass the guard across that
    await (TOCTOU) and both run the full download + read-modify-write install. The
    guard closes the common/manual case, not the wifiOnly background-overlap case.
  - The ledger fabrication hazard is concrete: cdr_daily --force is the exact
    mechanism (Section 2).
  - The failing mobile gates are likely shared-checkout WIP contamination, not
    merged main (P1-gates).


4. Critical Failures - Ranked by Evidence
-----------------------------------------

P0-ledger  Permanent ledger documented but not enforced.
  Evidence: [SOURCE] cdr_daily.py:99-137; cleanup_removed_cdr_sector.py:26-45,18-21.
  See Section 2. This is the governing constraint for all later work: the history
  data-plane fix (P0-history) MUST derive from, and never endanger, the ledger.

P0-history  Dashboard history is an unbounded bulk-row transport, not a data plane.
  Evidence: [MEASURED] + [SOURCE] cdr_dashboard_server.py:914-946 (bank_history_
  payload), :879-912 (read_bank_history_db), :929 (fill_history_gaps inflates
  rows), dashboard/app.js:194-218,276-293 (browser parse/normalize/index).
  Live Mortgage, 33 dates:
    Rows                 180,415
    Decoded JSON         113,659,904 bytes (~113.7 MB)
    Compressed transfer  5,227,322 bytes (~5.2 MB)
    First uncached req   ~22.3 s
    Cached gzip req      ~1.74 s
    JSON parse           ~519 ms (Node)
    Heap increase        ~352 MB
  The server reads rows from many SQLite files, materializes dicts, CREATES extra
  rows to fill gaps, serializes the whole result, and caches raw+gzip. The browser
  then parses, retains, and indexes all of it. At the 90-day read window this trends
  toward ~310 MB JSON / ~1 GB client heap - device-failure territory.
  Aggravators: [SOURCE] the history cache is ENTRY-count bounded
  (MAX_HISTORY_CACHE_ENTRIES, :942-945), not byte-bounded - it can retain ~1 GB at
  current sizes; and payload GENERATION happens OUTSIDE the cache lock (:922-939)
  with no per-key singleflight, so concurrent cold requests duplicate the full
  scan+serialize. This is the wrong data contract, not an optimization opportunity.

P0-ribbon  Live, user-visible ribbon-metric divergence (correctness bug).
  Evidence: [SOURCE] app_payload.py:168-197 aggregates via _effective_rate ->
  comparison_rate (fees folded in); cdr_dashboard_server.py:730-739 aggregates via
  row["rate"] (headline); dashboard/app.js:874 fetches /api/banks/ribbon and
  ribbonChartItems (app.js:498-535) renders the server's range/providers/counts
  VERBATIM. The comment at app_payload.py:135 still calls the builder a "faithful
  port"; it no longer is. For mortgages, comparison vs headline differ by the fee
  load (often tens of bps), so web and app show different "lowest rate" and
  different min/median/mean/max distributions for the same date+section TODAY. This
  is the cheapest high-value fix in the report: a ~10-line metric unification + one
  golden fixture. Ship it before any structural work.

P0-retry  CDR retry behavior amplifies upstream failures.
  Evidence: [MEASURED] + [SOURCE] cdr_ingest_support.py:449-509; cdr_ingest_lib.py:
  337-351. Six API versions are attempted; each gets its own full retry allotment
  (max_retries default 6 -> 7 attempts/version) -> 42 HTTP requests for one
  persistent-5xx logical fetch. A second fallback loop (:492-507) re-walks the
  version order and is near-dead in the common case (all versions already in
  `tried`) - defensive bloat. With default ~8 holder x 4 detail workers, a
  widespread outage can produce >1,300 concurrent-wave attempts; with timeout-style
  failures, 42 x ~90 s holds one logical request for over an hour. There is no
  shared retry/time budget, no circuit breaker, no Retry-After handling, and
  version fallback fires even on generic 5xx (which is not a version problem).

P0-fanout  Mobile product history does network fan-out + on-device compute.
  Evidence: [REPORTED] + [SOURCE] mobile/src/data/productHistory.ts:201-275;
  historyDaily.ts:114-130; store.ts:806-850. With the current 32-date index and a
  cold cache: 1 dates-index + up to 31 dated manifest + up to 31 dated core
  downloads ~= 63 requests for one feature, then on-device parse/aggregate across
  the catalogue (bounded worker pool of 3). The project already proved the correct
  pattern for section/bank history (precompute compact assets on the Pi - issue
  #0024); product history re-introduced the exact fan-out compact history removed.
  The current branch name (feat/pi-precompute-responsiveness) indicates this is
  already being addressed.

P1-cache  Mobile metadata updates rewrite the entire core bundle.
  Evidence: [SOURCE] mobile/src/data/cache.ts:155-195; callers at store.ts:540,
  597,633,765. readMeta() parses the whole bundle; updateMeta() reads it again and
  JSON.stringify(b.core) to atomically rewrite the entire ~7 MB core for a few
  metadata fields. Four optional-asset install paths call it -> up to ~28 MB of
  redundant writes plus repeated large parse/stringify per sync cycle. The
  core+meta coupling was a deliberate atomicity choice (store.ts:433); the fix is a
  version directory + atomic active-version pointer so metadata is a sub-KB write.

P1-dualbuild  Payload publishing rebuilds and rescans history twice.
  Evidence: [REPORTED] + [SOURCE] app_payload.py:621-703 (build_payload),
  :1361-1413 (build_and_publish_dual); app_payload_mobile.py:166-247. The dual
  publisher builds the dated payload, then builds the rolling payload again; each
  build reparses current exports and scans historical banks.json snapshots. Latest
  local banks.json ~34 MB; x ~32 history dates -> order-of-magnitude gigabytes of
  duplicate JSON read/parse per daily run on the Pi. Also [SOURCE] _published_
  history_dates (app_payload.py:869) is an N+1: list tags, then one manifest GET
  per dated release.

P1-contracts  Core financial semantics have parallel implementations.
  Evidence: [SOURCE] app_payload.py:135-223; cdr_dashboard_server.py:709-810;
  cdr_ribbon_normalize.py; mobile selectors/qualifiers. The same rate exists as raw
  ingest JSON, SQLite rows, section rows, history rows, mobile core rows, detail
  rows, search rows, and several normalized payloads. P0-ribbon is the live symptom;
  the disease is that no single module defines what "headline / comparison /
  ongoing-base / effective" mean, so each consumer re-interprets and silently drifts.

P1-gates  Current mobile branch fails its own quality gates.
  Evidence: [STATE] npm run typecheck -> 3 typed-route errors; npm run lint
  --max-warnings=0 -> 26 warnings (incl. missing React hook deps). Python tests
  63 passed; focused mobile history tests 24 passed; Pi smoke passed.
  CAVEAT: this is a known shared agent worktree with concurrent WIP (gitStatus
  shows M files; prior cross-agent contamination is documented in project history).
  Verify these failures against a CLEAN main checkout before counting them against
  committed code - the failures are real but their attribution is not established.


5. Systemic AI-Code Patterns Detected
-------------------------------------
Closure-trapped logic forcing copy-paste (root cause). ~30 functions nested in
  make_handler (cdr_dashboard_server.py:628) cannot be imported, so the ribbon /
  stats / normalize kernel was duplicated into app_payload.py:137-223. This is the
  mechanical cause of P0-ribbon and P1-contracts.

Feature-shaped accretion. Each optional mobile asset adds another cache file, SHA
  field, ensure* method, error field, retry path, validator, and persistence step.
  The store + cache metadata model expands linearly with every feature instead of
  through a stable install abstraction.

Fallbacks preserved after replacement. Compact history was added but the daily
  fan-out remains as a fallback, and the fallback can re-try the same failed
  compact asset. New path added without deleting/constraining the old one.

Parallel representations without a canonical contract (see P1-contracts).

Monoliths used as a coordination mechanism. The four oversized modules are large
  because they own too many policies (routing + SQL + aggregation + caching + gzip +
  proxy + HTML injection; transform + branding + crypto + packaging + publishing +
  pruning + CLI), not because the domain requires large files.

Tests without resource budgets. Tests mock network/cache and assert outputs and
  error order, but nothing fails when a feature issues 63 requests, writes 28 MB of
  redundant data, or returns 113 MB of JSON.

Operational automation outgrowing its source of truth. Many rules/skills/scripts/
  GitHub workflows, while WORKFLOW.md is empty despite being called authoritative.


6. Performance & Latency Risk Report
------------------------------------
Area                     Mechanism (confirmed)                         Cost
-----------------------  --------------------------------------------  ------------------------
Dashboard history        Raw multi-DB row dump + gap-fill inflation    113.7 MB JSON / 352 MB heap
History server cache     Entry-count bound; gen outside lock           ~1 GB potential; dup CPU
Cold history requests    No per-key singleflight                       duplicate generation
Mobile product history   Per-date manifest/core fan-out                ~63 requests (cold)
Mobile metadata update   Full core parse/stringify/rewrite             ~28 MB redundant writes
CDR version fallback     Retry budget repeated per version             42 attempts/logical req
Dual payload publishing  Full build + history scan x2                  GB-scale duplicate Pi I/O
Dates discovery          Tag list + 1 manifest/date                    N+1 GitHub requests
Macro ingest             Source families run serially                  wall-clock = sum, not max

This is a Pi + GitHub-release system, so the damage is not cloud spend: it is user
latency, Pi CPU/RAM pressure, mobile battery/data, hostile load on upstream CDR
holders, and slow outage recovery. The transport layer itself is sound (XHR
progress, 30 s timeouts, SHA-before-cache, gzip). Every row above is misplaced or
repeated COMPUTATION, not a slow primitive.


7. API, Contract & Failure-Handling Risks
-----------------------------------------
  - Schema versioning exists but does not validate payload SHAPE; core, details,
    and search-index paths lean on casts (only bank-insights and history-banks are
    validated).
  - Optional/non-fatal work is not consistently separated from release-critical
    freshness: a successful ingest does not prove the mobile payload is fresh.
  - Broad exception handling preserves availability but lets partial datasets look
    successful unless an operator inspects secondary logs/manifests.
  - The dashboard API exposes STORAGE-shaped data (raw rows) rather than
    TASK-shaped contracts (windowed, identity-keyed series).
  - GitHub release publication is a sequence of CLI/API calls with N+1 manifest
    checks inside a 1,000+-line module - robust in places, hard to reason about.


8. Human-Led Repair Plan (staged, ledger-first)
-----------------------------------------------
Do NOT begin with routing tables, generic wrappers, or a rewrite - those are
distractions. Begin by making failure impossible to ship and the ledger impossible
to lose; then fix the expensive data paths; decompose last.

Phase 0 - Establish non-negotiable budgets (tests/probes that FAIL when):
  - 90-day Mortgage serving response > 2 MB gzip or > 20 MB decoded.
  - Cached history response > 250 ms on Pi; cold > 2 s on Pi.
  - Product history performs > 2 network requests (manifest + asset).
  - One CDR logical request > 8 total attempts across all versions.
  - Metadata-only cache update writes > 32 KB.
  - Mobile typecheck / lint / tests / export fail.

Phase 0A - Make the permanent ledger real (parallel with Phase 0):
  - Declare 2026-05-13 epoch; record 2026-05-14 as a known gap.
  - Finalize each daily partition with hashes, counts, source metadata, continuity
    linkage; reject later mutation, deletion, AND same-date overwrite.
  - Replicate finalized partitions off the Pi; keep one independently recoverable
    copy; schedule integrity verification, missing-day alerts, capacity forecasts,
    restore drills.
  - Put cleanup_removed_cdr_sector.py and any destructive utility behind audited
    break-glass controls.
  - Treat every serving window and cache limit as independent from ledger retention.

Phase 1 (correctness quick win, days not weeks) - Kill P0-ribbon:
  - Unify the server and builder ribbon aggregates on one explicit metric; add a
    golden fixture asserting web and mobile ribbons match for a known date+section.

Phase 2 - Replace the dashboard history data plane:
  - Define exact fields chart/hierarchy/provider-focus need; build a catalog over
    every immutable daily partition from epoch; precompute compact identity-keyed
    (or columnar) history artifacts FROM the ledger; query by window/identity, with
    "All" served via compact aggregates; add ETag/byte-budget caching + per-key
    singleflight; delete raw multi-DB assembly from the request path.

Phase 3 - Move product history to the Pi:
  - Emit regenerable product history in the existing single history pass; publish
    one content-addressed optional asset; mobile installs+validates; remove
    daily-core fan-out after a short compatibility window.

Phase 4 - Correct retry/outage behavior:
  - Per-holder version-capability cache; one shared retry/time budget across
    versions; global concurrency limit + holder circuit breakers; Retry-After;
    no version fallback on generic 5xx/timeout; expose incomplete-ingest status.

Phase 5 - Fix mobile persistence & ownership:
  - Split immutable core from mutable metadata (version dir + active-version
    pointer); in-flight promise dedup for ensure*; close the refresh-guard TOCTOU
    (set refreshing BEFORE awaiting onWifi, or use a single in-flight promise);
    move revision checks into a small dataset-install coordinator; split store.ts by
    ownership; add a state machine only where it removes real overlapping flags.

Phase 6 - Establish canonical contracts:
  - Pure financial metric/aggregation kernel with EXPLICIT metric selection
    (headline / comparison / ongoing-base / effective); generated/golden
    cross-language fixtures for manifest, core, details, history, qualifiers;
    validate every downloaded asset before install; remove duplicate transforms.

Phase 7 - Build payload/history once per ingest:
  - One canonical in-memory build + history result; package dated and rolling
    variants from it; persist incremental history state; make dates-index
    incremental (kill the N+1).

Phase 8 - Decompose monoliths by responsibility (LAST):
  - Boundaries: dashboard history service; banking section/ribbon service;
    static/proxy HTTP routing; payload transformation; payload packaging/
    encryption; GitHub release publication. Goal is ownership + testability, not a
    routing framework.

Phase 9 - Reduce workflow entropy:
  - Populate WORKFLOW.md or stop calling it authoritative; consolidate overlapping
    rules/scripts; keep ignored scratch/probe files out of the working root as
    local hygiene (already gitignored - not a shipped defect).


9. Final Verdict
----------------
Salvageable. Rewriting it would be wasteful. AR-local is not "AI hallucination
stitched into runtime": it contains substantial working domain logic and several
correctly engineered reliability mechanisms. It is also not production-ready in its
current architectural form.

Two architectural must-fixes dominate everything else:
  1. The permanent CDR ledger must be made technically durable, not merely intended
     (append-only enforcement, integrity chain, off-device backups, tested restore,
     deletion/overwrite guards). Until then the project's core promise - a complete
     historical record from 2026-05-13 - is one operator mistake or one "cleanup"
     refactor away from being false.
  2. The historical data plane must stop shipping raw rows to browsers and stop
     fanning out on devices; it must serve compact artifacts DERIVED from the
     ledger.

Highest-payoff sequence (note: this explicitly leads with the ledger and the cheap
ribbon correctness fix - both omitted from the ChatGPT draft's closing sequence):
  0. Lock in resource budgets AND make the ledger permanent (do both first).
  1. Ship the ribbon-metric fix + golden fixture (live correctness; days).
  2. Replace dashboard raw-history delivery (governed by the ledger invariant).
  3. Precompute product history on the Pi.
  4. Cap CDR retries globally + circuit-break.
  5. Separate mobile metadata from immutable core.
  6. Build payload/history once per ingest.
  7. Centralize financial-metric contracts.
  8. Decompose monoliths after the expensive paths are under control.

STOP adding historical features until the ledger is durable and Phases 2-3 are
complete. Every new visualization built on the current history contract compounds
the wrong architecture and increases the surface area that must remain consistent
with a ledger that is not yet guaranteed to endure.
