Ruthless Technical Audit — AR-local
Scope reviewed: Python ingest/payload/server (~15k LOC), mobile RN/TS app (~24k LOC), dashboard JS (~5.5k LOC). Findings are anchored to files/lines.

1. Executive Summary
This is not prototype junk, and it's not senior-designed either. It's an AI-generated skeleton that has been patched into reliability by genuinely competent hardening work — and the seam between the two is visible in every directory.

The hardening is real: SHA-pinned asset verification, atomic core+meta bundle writes, schema/min-version gating, a concurrency-limited worker pool, AES-GCM interop locked by cross-language test vectors. Someone who knows what they're doing has been here.

But underneath sit the textbook AI structural failures: the same domain math (rate normalization, ribbon aggregation, summary stats) implemented three-plus times across modules that can't share it; a 1,253-line HTTP server whose entire logic lives inside one make_handler closure dispatched by a 130-line if/elif waterfall; a mobile store that does read-modify-write across await boundaries and calls it state management. The projectmem issue log confirms it — a dozen open issues about "stale concurrent sync overwrite," "cache revalidation," "chart-gap" — these are the recurring symptoms of the structural debt nobody has paid down.

The codebase looks finished (exactly one TODO in the entire tree) while the issue ledger is full of open concurrency races. That gap — polished surface, unstable core — is the AI signature.

2. Critical Failures (Fix Immediately)
🔴 Triplicated domain math. _normalized_rate_value, _stats, aggregate_ribbon in the builder are re-implemented as normalized_rate_value, aggregate_ribbon_rows, stats in the server — and cdr_ribbon_normalize.py (764 LOC) is supposedly the canonical normalizer. Three sources of truth for how a rate becomes a number and how a ribbon is summarized. A definition change (e.g. how _effective_rate resolves bonus vs base) must be made in three places or the dashboard and the mobile payload silently disagree. Impact: correctness drift in the product's core numbers.
🔴 The server's logic is untestable. ~30 functions (DB reads, aggregation, payload assembly, caching) are nested inside make_handler, closing over export_resolver/site_root. You cannot unit-test aggregate_ribbon_rows without standing up the whole factory. This is why the math got copy-pasted into the builder instead of imported — it's trapped in a closure. The duplication is a symptom; this is the cause.
🔴 Store read-modify-write races. store.ts:409-451 reads get().core/get().details, awaits a multi-second downloadCore, then set()s and re-reads get().details. Two concurrent refreshes (scheduled + pull-to-refresh) interleave and clobber. The refreshing guard (store.ts:366) is a partial patch; meanwhile ensureDetails/ensureSearchIndex/ensureBankInsights are fired-and-forgotten with void (store.ts:357-364) with no in-flight ownership. This is open issues #0033/#0034.
🔴 Silent degradation everywhere. 309 try/except across 37 Python files (32 in app_payload.py, 32 in the server, 23 in cdr_macro_ingest.py). On mobile, per-date core failures (productHistory.ts:255-260) and the dates-index failure (productHistory.ts:208-215) are swallowed into debugLog.warn and execution continues. Failures don't surface as errors — they surface as "the chart is just missing some days," which is how issue #0007 ("ingest artifacts became incomplete after successful run") happens.
3. Systemic AI-Code Patterns Detected
Copy-paste DTO wrappers. payload.ts:214-303: downloadCore, downloadDetails, downloadSearchIndex, downloadBankInsights, downloadHistoryBanks — five functions with identical bodies differing only by JSON.parse(text) as T and an optional validator. This is one generic downloadJson<T>(url, sha, opts, validate?).
The if/elif router. route():1055-1184 linearly tests ~25 path strings per request, with a hardcoded JS asset allowlist (1064-1078) that must be hand-edited for every new file, and a bare except FileNotFoundError: pass fallthrough (1178) buried mid-chain with more branches after it. No dispatch table.
God modules. A 1,511-line builder of free functions; a 1,253-line server in one closure. No module ever exceeds the LLM's "one file, many functions" comfort zone into actual separation of concerns.
State-flag soup. status, refreshing, offline, refreshOutcome, payloadProgress, error — six overlapping booleans/enums modeling what is one refresh state machine.
The lesson learned, then re-violated. projectmem #0024 records replacing a "57-request daily fan-out" with a 1.38KB precomputed asset for ribbons. Then syncProductHistoryFromDailyPayloads re-introduces the exact same fan-out — downloading every dated core and recomputing on-device — for per-product history. (The current branch feat/pi-precompute-responsiveness is you discovering this for yourselves.)
False completeness. One TODO in the whole codebase, against an issue ledger of open races. Humans label their debt; AI hides it.
4. Performance & Latency Risk Report
On-device history fan-out (productHistory.ts): downloads N dated cores (worker pool of 3, L244-265) and recomputes best-rate-per-product across all sections, on a phone, on Pro unlock. Burns mobile battery and data, and re-does work the Pi already does for the section/bank assets. Move to a precomputed product-history asset — same single pass the builder already runs.
Hot-loop set-spreads (productHistory.ts:224-232): [...keys].some(...) allocated fresh inside a per-date loop, plus a [...keys].every catalog compare. O(dates × keys) with avoidable allocation. Pre-spread once. Minor in absolute terms, sloppy in principle.
Linear routing (route()): up to ~25 string comparisons per request. Negligible at single-Pi scale, but it's the structure that will bite the moment routes multiply.
The wins are real, credit where due: XHR progress + 150ms throttle, SHA verify before cache-under-hash, atomic bundle write, 30s timeouts, schema gating. The latency problem is architectural placement of computation (client vs Pi), not the transport layer.
5. API & Integration Inefficiency
Five redundant JSON.parse(...) as T transforms (see §3) — collapse to one validated generic.
Two validators (normalizeBankInsightsPayload, normalizeHistoryBanksPayload) are wired; three other assets (core, details, searchIndex) are cast unchecked. Inconsistent contract enforcement — either everything gets a schema gate or you admit you don't have contracts.
The /api/economic-data/series mixed-mode proxy (serve-local-if-complete-else-proxy-upstream) is a reasonable migration shim — but it's exactly the kind of stateful branch that needs a test and a removal date, neither evident.
6. Maintainability & Long-Term Risk
Repo hygiene is a fire hazard. Root contains 14 .tmp-*.json (including a 44MB .tmp-banks-history.json and a 22MB mortgage dump), 10 _pi_*.py/.pi_*.py ad-hoc probes, three _wt-* git worktrees nested inside the repo, and scratch dirs (runs/, .ar-mobile-audit/, .ar-bot-wait/) carrying their own node_modules. WORKFLOW.md is 0 bytes. This is the clearest "autonomous agents operated here and never cleaned up" signal in the whole tree. Gitignore it or delete it — today.
Debuggability: good logging primitives (debugLog, degradation log), undermined by swallow-and-continue (§2).
Testability: mobile is testable (real __tests__). Python server domain logic is not, because of the closure (§2) — which is the root of everything else.
7. Refactor Strategy (Human-Led Repair Plan)
Priority order, each independently shippable:

Extract one ribbon/stats kernel. Pull normalized_rate_value / aggregate_ribbon / stats / effective_rate into a single module (extend cdr_ribbon_normalize.py, which already claims that role). Builder and server import it. Delete the two copies. One test suite. This alone removes the #1 correctness risk.
De-closure the server. Lift the ~30 nested functions in make_handler to module-level functions taking explicit (resolver, root) params; the closure keeps only thin route bindings. Now the kernel from step 1 is reusable and the server is unit-testable.
Routing table. Replace the if/elif waterfall with a dict of (method, path) → handler plus a small prefix matcher for /assets, /site, /exports. Kills the hand-maintained JS allowlist and the mid-chain fallthrough.
Move per-product history to the Pi (your current branch). One precomputed product-history asset built in the existing pass; mobile drops syncProductHistoryFromDailyPayloads entirely. Net deletion of code and elimination of the on-device fan-out.
Collapse the download wrappers to downloadJson<T>(url, sha, opts, validate?) and apply a validator to all five assets, not two.
One refresh state machine in the store: a single refreshState discriminated union + a real mutex/in-flight promise that bootstrap/refresh/retry/ensure* all await. Stops the read-modify-write clobber at the source instead of patching symptoms PR by PR.
Repo hygiene: gitignore + purge the .tmp-*, _pi_*, _wt-*, and scratch dirs; either write WORKFLOW.md or delete it.
8. Final Verdict
Salvageable — and worth salvaging. Do not rewrite.

The hard, get-it-wrong-once stuff (crypto interop, atomic caching, payload integrity, offline-first sync) is already done competently. What's rotten is structural placement: domain math duplicated because it's closure-trapped, computation run on the client because nobody moved it back to the Pi, and state managed by flag-juggling because there's no state machine. Every one of those is a bounded, mechanical refactor, not a redesign — steps 1-3 are a focused week and retire the entire class of "the numbers don't match" / "concurrent sync clobbered my cache" bugs that the issue log keeps re-opening.

This is AI-generated bones with senior-grade muscle bolted on. Fix the skeleton and it's a genuinely solid single-operator product. Keep patching symptoms and you'll be re-litigating issues #0033/#0034 every quarter.

The duplicated ribbon kernel (step 1) and the on-device fan-out (step 4) are the two with the highest correctness/cost payoff. Want me to start on either — or log these as projectmem issues via the CLI?
